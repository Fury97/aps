UML：Unified Modeling Language统一建模语言

  • What is class and what is instance, what is object?
    • Object is an entity, no mater it’s a real thing or just a concept. It’s a package of attribute, data and operation.
    • Class is a collection of objects having the same or similar attributes. It is an abstract description.
    • An instance is an object of the class.
  • What is Abstraction?
    • Abstraction is defining the characters of things. Trying to distinguish one thing from others.
    • Abstract class can not be implemented directly.
  • 封装：Encapsulation
    • Hide unnecessary informations from outside user.
  • Visibility     可见性
    • public   +
    • private   -
    • protected   # 
    • package   ~
  • 面向对象编程三大特性
    • 多态，封装，继承
    • Polymorphism, Encapsulation, inheritance
  • Polymorphism     多态性
    • Polymorphism represents an interface can have different kinds of implementations. Like an abstract animal can have the implementation of cat, dog, duck. 
    • The benefit of polymorphism is that we can put several classes into a single super class, or abstract class. We can write a universal code suit for all these subclass. It avoids redundant codes and is easy to maintain.
  • Aggregation     聚合
    • We can aggregate two class or component together. This means both of them are part of something.
  • Coupling     耦合
    • Coupling describes how much two components depends on each other. 
  • Cohesion     内聚
    • When we encapsulate data, attributes and operations into a component, we call it cohesion.
    • When designing system, we tends to follow the rule of “High cohesion, Low coupling"
  • 实例    Instance    实例化     Instantiate
    • A independent entity with identity. Object is an instance of class.
  • 多重继承     Multiple Inheritance
  • Relationship in UML   UML中的关系
    • 依赖，泛化，关联，实现
    • dependency, generalization, association, realization

Dependency  依赖
It happens when:
  • A class needs message to or from another class

设计模式过筛
Lazy Initialization    延迟加载
Only when an object is used then it is initialized. This would the blocking when a bundle of objet is initialized together and block the system, or maybe in android the UI thread.

工厂模式  Factory Method Pattern
When the work of creating an object is heavy, or may be used frequently, or the programmer want this process be hidden from its caller, you can use a factory method, rather than calling its constructor. Factory method can also create an unspecified object, which can be plumped after the invoking of factory method.

单例模式    Singleton Pattern
With singleton pattern, the instantiation of a class is restricted to one object. When the cooperation among systems on a single object is need, singleton pattern can be used. 

适配器模式      Adapter Pattern
It can be used when you want to reuse some snippet of existing code, we can use adapter pattern to fit existing code with new codes.
It is done by convert an interface of a class into another interface clients expect.

桥接   Bridge
Decouple an abstraction and its implementation so that they can vary independently.

代理模式   Proxy
Proxy is a wrapper of an object, hiding its implementation. Caller don’t know and don’t care how indeed the job is done. It authorized its job to the proxy, proxy find the proper real implementation, preserving the interface calling method, and delegate the job to the real class.

装饰模式    Decorator Pattern
When you want do dynamically add some function into a class without statically inherit, you can use a decorator pattern. Client just call the decorator, decorator pass all methods to the concrete-decorator. After finish all methods, decorator can add some more operations to finish the decorating.
Decorator pattern don’t change the interface and only provide operation.

外观模式     Facade Pattern
When a system has a complicated appearance, which i mean a set of interface, this may confuse client, and it’s not easy for client to maintain its code. So we can create a higher-level class to perform a unified interface for client and hide the complex interface calling into this facade. This decouple the system in some degree.


策略模式    Strategy Pattern
When a class has different algorithm for different data or circumstances, we can use lots of if…else…, however this hardcoding is much rigid. We can dissociate the algorithm out to a separate class or interface, and use concrete subclass implement the interface. When client invoke the interface, client only have to call interface method, the strategy method determines which concrete subclass to call. The context provided by client is separated from algorithm

模板模式   Template Pattern
Let’s say we have a class contains a complex algorithm. When this class is apply in specified circumstance, some part of the algorithm needs to be substituted. We can use template pattern, finish the skeleton of the algorithm in the template class, leave the part to be fulfilled as abstract, then force subclass to finish that part, without changing the template. This greatly reduced the pressure of user and hide the part of data needed to be hidden. 例如一个排序算法可以把主代码完成在模板中然后仅仅把比较部分放出来抽象。
